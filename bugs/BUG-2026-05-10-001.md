# Bug Record: BUG-2026-05-10-001

## Summary

- Title: YouTube connection check runs before Flutter web Convex auth is ready
- Current status: `fix-attempted`
- Severity: `high`
- Owner: Diane
- Next command: `/sf-test --retest BUG-2026-05-10-001`

## Reproduction

- Preconditions:
  - Flutter web app is running with `CONVEX_URL` and Clerk configured.
  - A user reaches a feature route that watches `youtubeConnectionProvider`.
- Steps:
  1. Open a YouTube-dependent route such as `/playlists`, `/videos`, `/notes`, or `/play`.
  2. Let the app check `youtube:getYoutubeConnectionStatus`.
  3. Trigger an HTTP bridge failure or missing auth-token state during the check.

## Expected Behavior

- TubeFlow checks the YouTube connection only on YouTube-dependent surfaces and only after Clerk/Convex auth is ready. Google sign-in returns through `/sso-callback`, creates a Clerk app session, and then redirects inside the app. If YouTube is not connected, protected pages remain usable, non-YouTube pages do not run YouTube checks, and YouTube pages display a connect state instead of loading indefinitely.

## Observed Behavior

- Initial report: the web app reported `Failed to check YouTube connection: Bad state: ConvexClient not initialized. Call ConvexClient.initialize() first.`
- Retest report after the first fix: the web app still blocked all YouTube-dependent pages, including preferences navigation, with `Failed to check YouTube connection: Bad state: Missing Convex auth token for HTTP query bridge (youtube:getYoutubeConnectionStatus).`
- Retest report after the second fix: the app still loaded indefinitely while checking the YouTube library and did not allow normal navigation to settings while the user was not connected to YouTube.
- Retest report after the third fix: Google sign-in can be started, but after Google returns the user lands on `/#/sign-in?tf_redirect=/videos`; Clerk remains signed out and the bridge log shows no `handleOAuthRedirect` callback execution.
- Retest report after adding `/sso-callback`: build `b2290ff8118a0eaf2b0832e538dc9a566098053a` still lands on `/videos#/sign-in?tf_redirect=/videos`, suggesting ClerkJS is using the newer Core 3 redirect parameter names rather than sending the browser to the callback URL.

## Evidence

- User report in chat, no secrets or personal data included.
- User retest report in chat, no secrets or personal data included.
- User provided client logs showing repeated Clerk web `/v1/client` normalization messages and `preferencesDataProvider start`.
- Code evidence: `ConvexService.initialize()` skips `ConvexClient.initialize()` on web, while `query`, `mutate`, and `action` fell back from the web HTTP bridge to `ConvexClient.instance`.
- Code evidence: `youtubeConnectionProvider` called `youtube:getYoutubeConnectionStatus` unconditionally, while other auth-sensitive providers already guard with `waitForConvexTokenReady()`.
- Code evidence: `AppShell` mounted `YoutubeConnectBanner` on every protected route, so non-YouTube pages like Preferences still triggered `youtubeConnectionProvider`.
- Code evidence: `/sso-callback` was not a public GoRouter route, so if the router handled the callback before the service completed Clerk redirect handling, it could redirect back to `/sign-in` and lose the OAuth callback.
